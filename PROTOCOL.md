# Protocol — AgentSearchBench

**This document is preregistered.** It is frozen, hashed and pushed to GitHub
**before** the first question is generated. Its SHA-256 is recorded in each
run's `MANIFEST.json`. If we modified it after seeing results, the git history
would show it, and the run's manifest would point at the old hash.

That is the only guarantee that holds: we publicly give up, in advance, the
right to change the rules.

---

## 1. The question being asked

A search API for agents is not a search engine. Its customer is an LLM, and
that LLM pays **per token**. So the question is not "who returns the right
links", but:

> **At a given token budget, which API gives an LLM the material to answer
> correctly?**

Two axes, published together and never separately: **correctness** and **token
cost**. One number without the other is marketing.

## 2. The arms

| arm | what it is | config |
|---|---|---|
| **mako** | SERPdive, Mako model | default, no parameters |
| **tavily** | Tavily Search | `search_depth: basic`, default, no parameters |

**Why Tavily Basic, and it alone.** Tavily is the category leader and Mako's
direct substitute: same promise, same price, same buyer. Tavily *advanced*
costs two credits, takes ~4 seconds and returns more tokens — it is not the
same product nor the same buyer, and comparing it to Mako would be dishonest in
our direction as well as theirs. Exa and Perplexity play in another category
(proprietary index with tens of thousands of tokens; an LLM layer on top).
Brave and Serper sell raw SERP.

**And the "you picked the only adversary you can beat" objection?** The duel is
*pairwise*: it does not scale to N arms without becoming a different benchmark
(absolute grading, different biases). So our answer is not to add arms, it is
to hand you the harness: **`lib/arms.mjs` takes one more arm in ten lines, and
the 1000 questions are published.** Add yours.

**Syntheses are out of scope, on both sides.** Tavily's `answer` field is
`null` by default and we do not enable it; our `verdict` field is removed from
the payload sent to the judge. We compare the **material brought back**, not
the summarizer. Our `extra_info` (Google's answer box) stays: it is not a
synthesis, it is web content, and it is a default output of the API.

**Nothing is removed from the adversary's payload.** Each API's response goes
to the judge **verbatim** (`await response.text()`) — including Tavily's
`score` field, which their documentation presents as a ranking aid. No object
is rebuilt, no field is discarded.

## 3. The questions — written by third-party AIs, never by us

**1000 questions, 100 topics, 10 questions per topic.**

No question is written, edited, rephrased or removed by a SERPdive human. The
chain:

1. **The topics.** Each model of a panel of 5 LLMs from 5 different vendors
   (Anthropic, OpenAI, Google, xAI, DeepSeek) produces **in one call, without
   seeing the others, with no domain instruction whatsoever**, a list of 60
   candidate entities. The five lists are published (`candidates.json`). The
   harness **draws at random** 20 topics per list — and the seed of the draw
   is the SHA-256 of `candidates.json` itself: we can neither choose the seed
   nor touch a list without changing it. Anyone replaying the draw lands on
   the same topics. If duplicates and the news gate exhaust the decks before
   100, a **top-up round** is generated by the same mechanism — fresh lists
   published in `candidates<n>.json`, draw seeded by their own hash — repeated
   until 100. No human choice enters this refill.

   *Why a draw, and not round-robin picking?* We measured both. Sequential
   picking with an anti-duplicate list **anchors** the models on the domain of
   the topics already taken (massive drift toward a single theme); isolated
   calls without context return the same obvious entities (a third exact
   duplicates). Over-generation is the only tested variant where diversity
   emerges **without any instruction asking for it** — the models' own
   preference, published as is, drawn at random.
2. **The news gate.** Each drawn topic goes through `perplexity/sonar`, which
   summarizes the entity's recent news. If the summary describes no recent
   development **about the entity itself**, the candidate is skipped and the
   draw continues (same for a duplicate, by mechanical normalization). This is
   not steering: it is the definition of the test — a news-search API must be
   tested on news.
3. **The final duplicate check.** The mechanical rule misses cross-language
   duplicates ("Paris 2024 Olympics" / "JO Paris 2024"). A large model
   (`dedup_check` in the config) rereads **once** the numbered list of retained
   topics and flags entries that refer to the same entity. It can only **flag
   numbers** — never propose or choose a topic: replacements come from the
   draw, like everything else. Every verdict — news gate, mechanical rule,
   final check — is published in `gate.json`, sealed: the entire selection
   replays offline from `candidates.json` and `gate.json`.
4. **The questions.** Each panel model writes 2 questions per topic, from the
   summary. The model order **rotates** from one topic to the next, so that no
   model is systematically the first to take the salient facts.

The prompts are in `protocol/prompts/`, published verbatim. None contains a
style instruction: every line is justified either as a definition of the
product under test, or as a harness rule.

**Why 100 topics and not 10.** We measured that the dominant variance of a
bench is not judge noise — it is the **topic draw**: ±8 points. On an internal
run, Machu Picchu gave us 1-9 and an entertainment topic 8-2, with the same
code. A hundred topics crush that variance, and above all allow us to
**publish the score topic by topic, defeats included**.

## 4. The collection

Both arms are called **on the same day**, on the same question, without cache.
The raw HTTP response of each API is written as is into `payloads/<id>.json`.
Errors and timeouts are **kept and counted** (`error`), never silently
replayed, never removed from the denominator.

The collection is sequential and paced by a timer: a single question in
flight, a fixed pause between each. Spreading the requests over time keeps our
own calls from competing with each other and skewing the measurements
(latency, speed) — each request is measured alone on the wire, under the same
conditions for both arms.

The web moves: a run is **dated**. That is by design and published as such.
Replaying the collection tomorrow will return different pages — and the
protocol will be the same.

## 5. The judge

**Pairwise, blind, randomized.** The judge receives the question, an
**independent truth anchor** (an Exa search, a provider that is neither us nor
the adversary), and two payloads named **A** and **B**. It does not know which
is which. Mako sits in position A when the question index is even, B
otherwise — exactly every other question.

The judge does not grade style or length: the (published) instruction asks it
**which response enables a correct answer to the question**, leaning on the
anchor.

**The setup: one single judge, `openai/gpt-5.4`, over the 1000 questions.**
That is the published number, and there is only one.

**Why a single judge.** At n=1000, the standard error on a 60% win rate is 1.5
points: the **noise** is already averaged out, a second judge changes nothing
there. The only risk a panel would address is **systematic bias** — the idea
that the result is an artifact of gpt-5.4. But we are not asking you to
believe us on that: **the raw payloads are public**, and re-judging with
another model is a single command.

```bash
node bin/4-judge.mjs runs/<date> --model=anthropic/claude-sonnet-5
```

Your key, your model, our payloads. If our judge favored us, you would prove
it within the hour — and that guarantee is stronger than a second judge that
**we** would have chosen. We prefer to choose no additional arbiter at all.

## 6. What is published

Everything. `topics.json`, `questions.json`, every raw payload, every verdict
with its reasoning, `results.json`, the `MANIFEST.json` of hashes, and this
protocol.

**Two latencies, side by side, never one without the other.** The *wall*
latency is clocked by us around the HTTP call — same machine, same moment for
both arms, but it contains the network of the collecting machine. The *server*
latency is the one each vendor reports **itself** inside its payload
(`response_time_ms` for SERPdive, `response_time` for Tavily) — network
excluded, and without us ever clocking the adversary. Both are extracted from
the published raw payloads: recompute them.

`node bin/verify.mjs runs/<date>` recomputes the entire chain offline and
confirms that the published score is the arithmetic consequence of the
payloads.

## 7. What this protocol does not prove

**That the payloads are authentic.** No hash can prove that: a dishonest
vendor could fabricate its competitor's response. The countermeasure is not
cryptographic, it is social, and it rests on two points:

1. The questions are pushed to GitHub **before** the first payload. The commit
   timestamps attest to it. We cannot adjust the questions after seeing the
   results without rewriting the history — and that shows.
2. A Tavily key costs a few dollars. **Replay the collection.**
